# COGEval 平台契约总文档

本文是 `cogeval-platform-contracts` 仓库的中文契约入口。它索引当前由本仓维护的跨项目契约，并说明每类契约的生产者、消费者、机器校验件和详细文档位置。

## 总原则

- 本仓是 COG Eval 中台、Workbench 和其他消费端之间的跨项目 payload 契约权威来源。
- JSON Schema 和 Pydantic 模型是机器校验事实；文档解释契约语义，但不替代 schema。
- 所有跨项目 payload 必须携带 `schema` 字段。
- 生产者在发布、返回或持久化跨边界 payload 前应校验契约。
- 消费者在依赖外部 payload 前应校验契约。
- 夹具 fixture 是契约的一部分，生产者和消费者 CI 都应覆盖。

## 模型 thinking effort 配置

模型 thinking effort 配置仍由 `cogeval.provider_capability_catalog.v1` 承载，属于 Provider Capability Catalog 的模型级运行能力配置。

- 生产者：中台 provider 能力配置。
- 消费者：Workbench 执行配置。
- 参数面：在 `providers[].interfaces[].parameter_surfaces.thinking_effort` 中声明 provider adapter 实际传参路径、可选值和必要的值映射，例如 `reasoning.effort`、`generation_config.thinking_level` 或 `reasoning_effort`。
- 模型能力：在 `providers[].models[].capabilities.thinking_effort` 中声明该模型是否支持、Workbench 可展示/可下发的标准值、默认值、关联的 `parameter_surface` 和 `adapter_policy`。
- 回归夹具：`openai_reasoning.v1`、`anthropic_effort.v1`、`gemini_thinking.v1`、`deepseek_reasoning.v1` 覆盖不同 provider 的 thinking effort 配置形态。

## 迁移说明

原先散落在中台仓的工作台对接、账号资产、任务分发、COG Case 分组、网关一致性等契约说明，已经在本目录按契约领域归档。后续新增或修改跨项目契约时，应先更新本仓的 schema、fixture、测试和对应详细文档，再由中台或 Workbench 仓引用。

中台仓仍可保留架构背景、运营说明或历史设计文档，但跨项目 payload 的权威定义应以本仓为准。

## 新增契约流程

新增跨项目契约时，从 [ADDING_A_CONTRACT.md](ADDING_A_CONTRACT.md) 开始；版本兼容规则见 [COMPATIBILITY_POLICY.md](COMPATIBILITY_POLICY.md)；生产者/消费者 CI 建议见 [PRODUCER_CONSUMER_CI.md](PRODUCER_CONSUMER_CI.md)。
