# COGEval 平台契约总文档

本文是 `cogeval-platform-contracts` 仓库的中文契约入口。它索引当前由本仓维护的跨项目契约，并说明每类契约的生产者、消费者、机器校验件和详细文档位置。

## 总原则

- 本仓是 COG Eval 中台、Workbench 和其他消费端之间的跨项目 payload 契约权威来源。
- JSON Schema 和 Pydantic 模型是机器校验事实；文档解释契约语义，但不替代 schema。
- 所有跨项目 payload 必须携带 `schema` 字段。
- 生产者在发布、返回或持久化跨边界 payload 前应校验契约。
- 消费者在依赖外部 payload 前应校验契约。
- 夹具 fixture 是契约的一部分，生产者和消费者 CI 都应覆盖。

## 模型 thinking effort 配置

模型 thinking effort 的全局模型能力由 `cogeval.model_capability_catalog.v2` 承载；provider 可用性由 API Key Provider 配置承载；interface 参数面、interface 词汇和模型到 interface 的映射由 Model Capability Catalog v2 承载。

- 生产者：中台 provider 能力配置。
- 消费者：Workbench 执行配置。
- 模型值域：每个模型通过 `models[].thinking_effort.values[].id` 声明自己支持的 effort ID；该 ID 是稳定选择值、结果维度值，也是 UI 展示名。
- 参数面：`interfaces.<interface_id>.thinking_effort` 声明 interface 参数路径、interface 词汇和 wire value；`wire_value: null` 表示省略该参数。
- 接口模型能力：在 `models[].interface_capabilities.<interface>.thinking_effort.mapping` 中声明 model effort ID 到 interface vocabulary ID 的映射。
- 契约边界：v2 不再有 `platform_values`、`value_mapping`、`model_value_labels`，也不再有 model effort `display_name`；interface vocabulary 仍可有 `display_name` 作为接口词汇说明。
- Workbench 使用：Workbench 选择并保存 `models[].thinking_effort.values[].id`；运行时按 `model_id + model_effort_id -> interface_id + interface_vocab_id -> wire_value` 解析请求参数。
- 回归夹具：`minimal.v2.json` 覆盖 v2 model-owned effort ID、interface vocabulary 和映射形态。

## 迁移说明

原先散落在中台仓的工作台对接、账号资产、任务分发、COG Case 分组、网关一致性等契约说明，已经在本目录按契约领域归档。后续新增或修改跨项目契约时，应先更新本仓的 schema、fixture、测试和对应详细文档，再由中台或 Workbench 仓引用。

中台仓仍可保留架构背景、运营说明或历史设计文档，但跨项目 payload 的权威定义应以本仓为准。

## 新增契约流程

新增跨项目契约时，从 [ADDING_A_CONTRACT.md](ADDING_A_CONTRACT.md) 开始；版本兼容规则见 [COMPATIBILITY_POLICY.md](COMPATIBILITY_POLICY.md)；生产者/消费者 CI 建议见 [PRODUCER_CONSUMER_CI.md](PRODUCER_CONSUMER_CI.md)。
